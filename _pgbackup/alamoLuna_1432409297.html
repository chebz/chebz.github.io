Strangeioc

The main reason I've started working on this game was so I can learn StrangeIoC.

StrangeIoC is a relatively new hip framework based off Robotlegs for building decoupled and highly abstracted code.

It introduces two features: MVC (model-view-controller pattern) and IoC (inversion of control, dependency injection in particular).&nbsp;<br> 

You can read more about the ways of the Strange here

My favorite feature with Strange is that it breaks up scene objects under separate contexts. This provides better decoupling and encapsulation. It also facilitates code architecturing when thinking about it in terms of contexts.

In the example image you can see MainContext which is loaded up with game startup, it controls lifespan of subcontexts, such as IntroContext and MenuContext.

I also like to include state pattern to determine application state and decide which contexts should be loaded or unloaded (in this example ApplicationManager object under MainContext does the exact thing).

One of the major downsides to StrangeIoC is that the game is limited to a single scene (in case you prefer to use cross-context objects or events/signals). Which is a huge pain in the behind when new level needs to be loaded in the place of old level: all the objects need to be manually destroyed, all bindings manually unbound (and there seems to be some bug in Strange when trying to unbind cross-context bindings).

Another big argument I've had over use of StrangeIoC is how useful is MVC with game development? Unity already kind of handles that for you: serialized objects in the scene already have their "model" (properties) exposed in Unity editor where the view is the logic underneath. Is there a good reason to create a separate "model" class which cannot be as easily accessed through unity editor? More specifically:

1) When is decoupling to model-view actually needed? 

2) When should Monobehavior derive from View? 

3) Why the hell do we need Mediator pattern for every view as stated in the doc?

These thoughts are based purely on my experience working on this game:

1) A separate Model class can come in handy when a set of properties needs to be shared between views. For example, when connecting user interface event with the game code: a user performs an action in UI such as making a selection or inputing data, that action-specific data can then be neatly wrapped into a model and broadcasted via a signal. This allows to write properly encapsulated code. Another example is with use of Memento pattern: when object state needs to be stored/restored - its properties can then be contained separately in the model class.

2) Deriving from View class seems to be a very subjective choice. Obviously when class needs to listen to signals it needs to derive from View, otherwise if it's some utility or very generic class class that doesn't belong to any context in particular I think its better to leave it as is. The best practice in my opinion is to not make object a View unless absolutely necessary.

3) Mediators are used as a connector blocks between views and the rest of the application. They listen for signals and provide public access to common methods, sort of like extended interfaces. I never found them very useful. Most views have at most 3-4 signals, usually a lot less and there is really no point to delegate that to a separate Mediator class bloating the codebase with useless boilerplate code. 

CHICKEN OR EGG?

One of the interesting problems I've ran into (and very annoying one too!) was getting Strange to work with objects created at a run-time that inject other objects created at a runtime. If you look into the code for the View class, injected properties are set when Awake or Start functions are called. If there is no such object found to inject in the injectionbinder then an exception is thrown. Thus the biggest issue here is order of registering with context.

For example every enemy ship needs to be aware of the player. But what if there is no player in the scene whenever an enemy is spawned?
